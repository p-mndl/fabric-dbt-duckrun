Instructions for an AI coding agent (or a human) working on a fix proposal / PR for
[djouallah/duckrun#16](https://github.com/djouallah/duckrun/issues/16). Everything below was
established by static analysis of duckrun 0.4.21 (PyPI wheel) plus this project's setup; the
open verification steps are marked as such.

## Symptom

`dbt show -s <one small model>` against OneLake takes 30–35 s from a residential connection,
although the model's own SQL runs in 1–2 s. The delay happens **before** the selected node
executes. Project size at the time of measurement: ~50 models, 18 snapshots, plus the
elementary package (~15–20 extra models), spread over multiple schemas and two catalogs
(default/gold + attached silver). Environment: dbt-core 1.11.12, duckrun 0.4.21.
A reproducible fixture of the same scale lives in this repo under `dbt/models/perf/` +
`dbt/snapshots/perf/` (tag `perf_repro`, generated by `tools/generate_perf_repro.py`).

## Root cause (duckrun 0.4.21, verified by reading the source)

dbt populates its relation cache at task start for **every schema in the manifest** (default
behavior — not just the selected node's schemas). For each schema, duckrun's
`list_relations_without_caching` (`dbt/adapters/duckrun/impl.py`) does:

1. **One DFS "List Paths" REST call per schema** — `_discover_via_rest` →
   `remote.list_delta_tables` (`remote.py`), `recursive=false` on `<root>/<schema>`. Cheap.
2. **Per discovered table, a tombstone check** — `delta_dml.is_dropped` runs
   `select * from delta_scan('<loc>') limit 0`, which makes DuckDB's delta extension read the
   table's full `_delta_log` over HTTPS (log listing + `_last_checkpoint` + json/checkpoint
   fetches).
3. **Per table, view registration** — `_register_delta_view` executes
   `CREATE OR REPLACE VIEW ... AS SELECT * FROM delta_scan('<loc>')`; DuckDB binds the view at
   creation → a **second** full log read.
4. **Per table, comment restore** — `_apply_delta_comments` → `engine.read_delta_docs` opens
   the table again via **delta-rs** (`DeltaTable(path)`) → a **third** log read, through a
   separate HTTP client with no metadata cache shared with DuckDB.

All of this runs **sequentially**, for **every table in every manifest schema**, regardless of
what `-s` selected. Cost model: ~3 log opens/table × 3–6 HTTPS round-trips/open × ~80–100
tables ≈ 1000+ serial round-trips. At 30–100 ms per round-trip (residential → OneLake) that is
exactly the observed 30+ s. The problem is latency-bound: machine speed is irrelevant, and
inside a Fabric notebook (low latency to OneLake) the same work finishes much faster — which
is likely why upstream doesn't feel it.

Log fingerprints of the loop: repeated `duckdb_databases()` queries and
`create schema if not exists "<catalog>"."<schema>"` statements (from `create_schema` inside
`_register_delta_view`), plus `duckrun:`-prefixed debug messages (visible with `dbt --debug`).

## Verification status

Confirmed: code paths above (static analysis). Still to confirm empirically (in progress in
this repo, using the `perf_repro` fixture):

- `dbt --no-populate-cache show -s <model>` collapses the runtime → proves the time is in
  project-wide cache population (dbt then discovers lazily per referenced schema only).
- `dbt show -s <model> --cache-selected-only` → the realistic dbt-side mitigation.
- Removing elementary shrinks the time roughly by its share of total tables → elementary is an
  amplifier, not the cause.
- Single-table open timings (one `delta_scan(...) limit 0` and one delta-rs `DeltaTable(...)`
  open, measured standalone) × table count × ~3 should reproduce the baseline within ~2×.

If any of these contradict the model — especially if `--no-populate-cache` is still slow —
re-derive the bottleneck from a `dbt --debug` log (timestamp gaps) before designing a fix.

## Goal for a duckrun change

Make read-only commands (`show`/`test`/`docs generate`/`compile`) pay discovery cost
proportional to the **selected nodes' dependency graph**, not the whole project — without
breaking the correctness guarantees the current code deliberately provides:

- Drop-tombstones must never surface as live relations (`is_dropped` exists for this).
- Discovered Delta tables must be reported as `table`-typed relations so `is_incremental()`
  stays true across fresh processes (disk wins over stale in-memory views — see the merge
  logic at the end of `list_relations_without_caching`).
- A genuine access failure (`OneLakeAccessError`: wrong tenant, item not in workspace) must
  fail loud, never masquerade as an empty schema.
- Persisted Delta docs must still show up in `dbt docs generate` output.
- Read-only commands must still be able to query tables that no materialization created in
  this process (that's why views are registered during discovery at all).

## Proposed direction, in order of expected impact

1. **Lazy per-table registration (the actual fix).** During cache population, return the
   discovered relation names only (step 1 — the REST listing — is enough for that). Defer
   tombstone check, view creation, and comment restore until a relation is actually used:
   either hook relation access (e.g. override `get_relation`/`create_schema`-adjacent paths or
   register on first cache hit), or restrict eager registration to schemas/tables of selected
   nodes + their ancestors if the adapter can see the selection. Tombstone visibility can be
   handled at first access too (a tombstone that is never touched doesn't need hiding).
   Trade-off to document: a tombstoned table would appear in the relation cache by name; check
   whether anything user-visible (e.g. `dbt ls`, `docs`) exposes it and gate accordingly —
   `docs generate` legitimately needs the full eager path and can keep it.
2. **One log open instead of three.** For whatever per-table work remains eager: open the
   table once (delta-rs) and derive everything from that single open — schema for the
   tombstone check (`_columns_are_tombstone` needs only column names), docs from
   `dt.metadata()`/`dt.schema()`, and create the DuckDB view without a bind-time log read
   (e.g. create it lazily, or investigate a typed view definition that doesn't force
   delta_scan binding at CREATE time).
3. **djouallah's suggestion — one recursive DFS listing per catalog root** (the machinery
   already exists: `remote.list_files`). Replaces S per-schema listing calls with 1–2 calls
   per catalog and yields `_delta_log` file names for all tables as a bonus (usable to skip
   obviously-absent logs and possibly to short-cut tombstone/emptiness detection). On its own
   this is a 10–20 % win because listing is already the cheap part — implement it *with* (1)
   and (2), not instead of them. Mind DFS paging (5000 entries/page) on large lakehouses.
4. **Parallelize** the remaining per-table remote calls (thread pool over tables; the REST
   listing and delta-rs opens are independent per table).

dbt-side mitigations (`--no-populate-cache`, `--cache-selected-only`) are user workarounds and
useful benchmark baselines, but the adapter should be fast by default.

## Practicalities

- Source: https://github.com/djouallah/duckrun — the dbt adapter lives under
  `dbt/adapters/duckrun/` (impl.py, remote.py, delta_dml.py, engine.py, secret.py) and the
  wheel on PyPI matches the repo. Target the fix as a PR against that repo, referencing
  issue #16.
- Benchmark before/after with this repo's fixture: `dbt build -s tag:perf_repro` once, then
  time `dbt show -s dim_publicholidays` (2nd run of each variant; residential connection and
  Fabric notebook are both interesting data points). Include the numbers in the PR.
- Keep the existing best-effort error handling philosophy: discovery failures degrade
  gracefully (log at debug), except `OneLakeAccessError` which stays loud.
- duckrun has tests that stub the REST layer (`_sleep` indirection etc. suggest a test suite);
  extend them for the lazy path rather than only integration-testing against OneLake.
